# 岗位检测机制 & 递进范式设计

---

## 一、岗位怎么检测

### 1.1 三级优先级

```
求职意向栏提取 → 最高优先级（简历中写了"求职：后端开发"直接用）
    ↓ 没写
简历+JD关键词扫描 → 自动兜底（统计命中量最高的岗位）
    ↓ 置信度低
用户手动在下拉栏选择 → 最终兜底（设置页"岗位 & JD"面板）
```

### 1.2 关键词匹配引擎

每个岗位有一个关键词库，扫描简历全文+JD，统计命中量：

| 岗位 | 关键词数 | 样本 |
|------|---------|------|
| 后端开发 | 18 个 | 后端、Java、Go、MySQL、Redis、Kafka、分布式、微服务、并发、RPC、API、数据库、架构 |
| 前端开发 | 18 个 | React、Vue、TypeScript、Webpack、浏览器渲染、跨端、工程化、防抖节流、小程序 |
| 产品经理 | 15 个 | 产品经理、需求、原型、Axure、用户调研、AB测试、留存、转化、竞品分析 |
| 数据分析 | 15 个 | 数据分析、SQL、Tableau、指标体系、埋点、数据仓库、统计学、Hive、Spark |
| 运营 | 14 个 | 运营、活动策划、渠道投放、用户留存、社群运营、转化数据、流量渠道、KOL |

**算法**：遍历所有关键词 → 统计每个岗位的命中数 → 选最高分的岗位。分数 < 3 为低置信度，提示用户手动指定。

### 1.3 当前缺失：求职意向提取

简历解析 Prompt 目前不提取"求职意向"栏。需要加 `jobIntent` 字段。加上后，自动检测的优先级变为：

```
求职意向 = "后端开发" → 直接使用，跳过关键词匹配
求职意向 = 空 → 走关键词匹配
关键词匹配置信度低 → 提示用户手动选
```

### 1.4 手动选择 & 自动填写

设置页「岗位 & JD」面板中，下拉栏默认选"自动检测"。如果从简历中提取到了求职意向，自动填入对应选项并提示"已从简历中检测到"。用户可随时覆盖。

## 二、每个岗位的递进范式

### 设计原则

四层递进的**结构相同**——都是概述→细节→难点→量化，但每层的**提问内容**由该岗位的核心能力决定：

| 层 | 决定因素 | 每个岗位问什么 |
|----|---------|-------------|
| 第 1 层（概述） | 该岗位最常问的项目背景 | 后端问架构/模块，产品问需求背景，运营问活动目标 |
| 第 2 层（细节） | 该岗位最核心的硬技能 | 后端问技术选型，产品问用户调研，运营问投放策略 |
| 第 3 层（难点） | 该岗位最典型的压力和冲突 | 后端问高并发，产品问跨部门冲突，运营问ROI过低 |
| 第 4 层（量化） | 该岗位最关注的结果指标 | 后端问QPS/延迟，产品问DAU/留存，运营问转化率/ROI |

### 五个岗位完整范式（Prompt 原文）

#### 后端开发

```
1. 概述：负责的模块与架构全貌
   → "这个项目你负责了哪些模块？架构大概什么样？"

2. 细节：技术选型、性能指标（QPS/延迟）、缓存/数据库方案
   → "为什么用Redis而不是本地缓存？数据一致性怎么保证的？"

3. 难点：最大技术挑战、trade-off；必问高并发/故障/扩容极端场景
   → "如果流量涨10倍，架构哪里会先扛不住？"

4. 量化：具体性能数据和压测结果，重做如何改进
   → "优化后QPS从多少涨到多少？怎么压测验证的？"
```

**设计重点**：后端面试最看重技术深度和系统设计。第二层追"为什么这样选"，第三层上极端场景——区分"会用工具"和"真懂原理"。

#### 前端开发

```
1. 概述：负责的前端模块、业务场景、技术栈（React/Vue/小程序）
   → "这个项目你负责了哪些页面和组件？"

2. 细节：打包优化、渲染优化、跨端适配、组件设计、埋点实现
   → "Webpack配置你怎么优化的？首屏从多少降到多少？"

3. 难点：技术选型对比、浏览器兼容、线上渲染故障、组件库设计
   → "有没有遇到过线上白屏？排查思路是什么？"

4. 极端场景：超大列表卡顿、弱网适配、百万并发页面性能、多端冲突
   → "列表10万条数据渲染卡了怎么办？"
```

**设计重点**：前端面工程化和性能。第三层追故障排查——线上问题直接影响用户，排查能力是关键。

#### 产品经理

```
1. 概述：需求背景、个人角色
   → "这个需求的背景是什么？你在项目中的角色？"

2. 细节：需求确定过程、用户调研方法、数据驱动决策
   → "需求怎么确定的？做了哪些用户调研？有数据验证吗？"

3. 难点：被砍需求、多方冲突（研发不配合/资源被砍/需求变更）
   → "有没有研发说做不了的？你怎么推的？"

4. 量化：功能上线后核心指标、复盘改进点
   → "上线后数据怎么样？DAU/留存有变化吗？如果重做怎么改？"
```

**设计重点**：产品面试最看重"怎么确定要做什么"和"怎么推得动"。第二层验证需求来源是否靠谱，第三层是核心区分度——能推得动的人远比会画原型的人值钱。

#### 数据分析

```
1. 概述：业务目标、负责环节
   → "这个分析项目的业务目标是什么？你负责哪个环节？"

2. 细节：数据来源、分析方法/模型选择、技术实现
   → "数据从哪里取的？为什么选这个方法而不是其他？"

3. 难点：数据质量问题、结论被质疑的应对；必问数据可信度场景
   → "分析结论被业务方质疑过吗？你怎么应对的？"

4. 量化：分析推动的业务决策、产生的业务价值和ROI
   → "这个分析最终推动了什么决策？产生了多少业务价值？"
```

**设计重点**：数据分析面试最怕"只会跑数不会思考"。第三层追问"结论被质疑"——真做过的分析师一定遇到过，能讲出具体应对方案的才叫懂。

#### 运营

```
1. 概述：负责的活动/渠道/用户运营工作与核心业务目标
   → "这个活动/渠道的核心目标是什么？你在其中的角色？"

2. 细节：活动策划流程、渠道投放策略、转化指标设计、用户分层
   → "活动策划的具体流程？投放渠道怎么选的？转化漏斗怎么搭的？"

3. 难点：效果不达预期复盘、ROI过低优化、用户流失、跨部门冲突
   → "有没有活动效果远不如预期的？你怎么复盘的？"

4. 极端场景：预算减半怎么提升转化、零流量冷启动破局、突发风控/投诉
   → "预算突然砍了一半，你怎么调整策略保证转化？"
```

**设计重点**：运营面试注重"结果导向"和"复盘能力"。第三层专门追问"没达标"的情况——好的运营能从失败中提炼方法论。

## 三、范式设计的通用规律

### 规律 1：第 3 层是区分度最高的层

第 1、2 层只要参与过项目都能回答。第 3 层的极端场景/冲突场景，没真做过的很难编。

### 规律 2：每层的追问语言要和岗位匹配

- 后端追问用"怎么保证""怎么验证"（技术验证思维）
- 产品追问用"怎么确定""怎么推动"（决策和协作思维）
- 运营追问用"怎么复盘""怎么调整"（迭代优化思维）

### 规律 3：第四层必须留下"如果重做"的空间

追问"如果重做会怎么改进"不是为了刁难，是为了看候选人有没有反思能力——这是校招中最稀缺的特质。
