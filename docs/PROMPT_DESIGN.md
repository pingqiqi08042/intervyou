# IntervYOU Prompt 系统设计文档

---

## 一、整体架构

```
用户选择：岗位（自动/手动）+ 模式（引导/标准/压力）+ JD（可选）
                        ↓
              buildSystemPromptV2()
                        ↓
          ┌─────────────┼─────────────┐
          ▼              ▼              ▼
     岗位检测引擎    三模式定义    四层递进框架
   (detectJobRole)  (MODE_DEFS)  (FOUR_LAYER)
          │              │              │
          └──────────────┼──────────────┘
                         ▼
                  完整 System Prompt（~1400 tokens）
                         │
                         ▼
                  发送给 DeepSeek
                         │
                         ▼
              Agent 返回 JSON 响应
```

## 二、Prompt 六大模块

### 模块 1：最优先规则（Top of Prompt）

```
1. 一次只问一个问题
2. 绝不允许重复提问 —— 检查上一轮问了什么，这一轮必须不同
3. 候选人给了 3 句以上详细回答 → 必须进入更深层或换题
4. 每轮回复前自查：我刚才问的是第几层？这轮必须推进
5. 绝对禁止回退到概述层
```

**设计意图**：把最重要的约束放在 Prompt 开头。LLM 对开头的注意力权重最高，后面的内容可能被忽略。

### 模块 2：岗位机制

**5 类岗位识别流程**：

```
用户输入"岗位=XX" → 同义词归一（后端工程师→后端开发）
                  → 白名单校验（不在白名单→提示重新输入）
                  → 生效，切换提问范式

无手动输入 → 关键词检测（简历+JD 文本）
           → 置信度分 3 档（低/中/高）
           → 低置信度 → 提示用户手动指定
```

**每岗位 4 层递进**：

| 岗位 | 第 3 层（难点）专属场景 |
|------|----------------------|
| 后端开发 | 流量涨 10 倍架构怎么变 |
| 前端开发 | 超大列表卡顿、弱网适配 |
| 产品经理 | 研发不配合、资源被砍 |
| 数据分析 | 结论被质疑、数据质量 |
| 运营 | ROI 过低、预算减半 |

### 模块 3：三模式差异化

```
引导模式：每层 1 问 + 缓冲语 + 答不上给提示 + 评分宽松
标准模式：每层 1-2 问 + 中立专业 + 答不上记录弱点
压力模式：每层 2-3 问 + 犀利质疑 + "太虚了，具体数字呢？"
```

实际回复对比：
- 引导："没关系，慢慢想。比如你们当时第一个实验是怎么设计的？"
- 标准："了解了。那实验的具体设计流程是怎样的？"
- 压力："太虚了。具体数字呢？DAU 从多少涨到多少？我要听指标和逻辑。"

### 模块 4：四层递进框架

```
第一层（概述，1 问）
    ↓ 候选人回答了 → 进入第二层
第二层（细节，1-2 问）
    ↓ 追问技术/方法/决策具体落地
第三层（难点，1-2 问）
    ↓ 追问困难、trade-off、极端场景
第四层（量化反思，1 问）
    ↓ 追问结果、数据、沉淀
换题 → 点名下一段简历经历
```

**关键规则**：每层只能问一次，绝不回退。

### 模块 5：换题规则

```
强制换题信号：
- 候选人说"不是我做的" → 立刻换
- 四层深挖完毕 → 换下一段经历
- 连续两轮敷衍 → 换

换题格式：
✅ "了解了。你简历上还写了XX在网易的实习，聊聊那个吧。"
❌ "我们换个话题。"（不知道你想聊什么）
```

### 模块 6：JSON 输出格式

```json
{
  "message": "面试官说的话（40-120字）",
  "metadata": {
    "evaluation": { "starCompleteness": 3, "quantification": 3, ... },
    "phase": "resume_deep",
    "questionIndex": 5
  },
  "isComplete": false
}
```

## 三、服务端防护层

除了 Prompt 约束，服务端还有两道防线：

### 防线 1：历史脱敏（agent.ts）

```
每轮发送前，解析历史消息中的 JSON
→ 只保留 message 字段（面试官说的话）
→ 丢弃所有 metadata（评分、notes）
→ LLM 看不到自己之前的评分
```

### 防线 2：重复检测（chat route）

```
Agent 的新回复 vs 历史消息：
→ 完全相同 OR 前 15 字相同 OR 包含"负责了哪些""展开说说"
→ 拦截，自动替换为点名具体公司的换题语
```

## 四、为什么还会出现重复提问？

**根因在于 DeepSeek V3 的"指令遵循漂移"**。Prompt 在最顶部写了"绝不允许重复"，但随着对话轮次增加，DeepSeek 对早期 Token 的注意力衰减，后面的规则可能被忽略。

**当前缓解措施**：
1. 最优先规则放在 Prompt 开头（注意力最高）
2. 规则简洁、口语化（"这是面试不合格的表现"比"严禁重复提问"更有效）
3. 服务端拦截（兜底）

**如果追求彻底解决**：换 Claude——Claude 的指令遵循能力比 DeepSeek 强，但成本高 20 倍。

## 五、Prompt 迭代历史

| 版本 | 关键改动 | 效果 |
|------|---------|------|
| V1 | 基础 Prompt 骨架 | 追问重复率 ~20% |
| V2 | 加"强制换题信号"+ 防重复规则 | 重复率降到 5% |
| V3 | 开 JSON mode + 历史脱敏 + 降到 Temp 0.5 | JSON 错误率近零，Token -53% |
| V4 | 岗位自动识别 + 四层递进 + 三模式强化 + 禁止回退 | 17/17 测试通过 |

**核心发现**：
1. Temperature 0.7 对 JSON 模式太高 → 0.5 刚好
2. 评分 JSON 回传给 LLM 导致行为锚定 → 脱敏后显著改善
3. Prompt 不是越长越好 → 3600→1400 tokens 后质量反而上升
4. 规则放在 Prompt 最顶部比底部有效得多
