# Prompt 调优总结

> 日期：2026-07-11 | 模型：DeepSeek V3 (deepseek-chat) | 测试用例：14 个

---

## 一、做了什么

将 IntervYOU 面试 Agent 的 System Prompt 工程化为**可测试、可对比、可迭代的模块化系统**。经过 4 轮调优，14 个测试用例全部通过。

### 模块架构

```
┌──────────────────────────────────────────────┐
│                buildSystemPrompt()            │
│                  组装最终 Prompt               │
├──────────────────────────────────────────────┤
│  base.ts      角色人格、追问框架、换题信号     │
│  modes.ts     4种面试模式的行为差异            │
│  difficulty.ts 3级难度的语气差异               │
│  evaluation.ts 输出格式 + 内部评估 + 复盘      │
│  resume-parser.ts 简历解析 prompt             │
└──────────────────────────────────────────────┘
```

### 核心设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| LLM Provider | **DeepSeek V3** | 便宜 20-50 倍，中文好，OpenAI 兼容 |
| 输出格式 | **JSON mode**（response_format） | 解决 DeepSeek JSON 不稳定问题 |
| 评估策略 | **评估与对话分离** | 面试中内部打分→面试后一次性释放复盘 |
| 追问引擎 | **基于简历的决策树** | 不靠题库，从简历出发动态追问 |
| Prompt 组织 | **模块化 + builder 组装** | 改一个维度不影响其他，测试独立 |

## 二、四轮调优过程

### 轮次 0：初始设计（跑通骨架）

写了 base、modes、difficulty、evaluation 四个模块，8 个测试用例。
结果：8/8 自动检查通过，但人工审查发现两个问题。

### 轮次 1：修复核心行为问题

**问题 A — 候选人说"不是我做的"后 Agent 继续追问**

修改 `base.ts`：在追问框架中增加"强制换题信号"——

```
以下情况必须立即换题：
1. 候选人承认不是自己做的 → 直接过渡到下一段经历
2. 连续两轮答不上来 → 记录弱点，换题
3. 回答在同一个话题上原地打转 → 水位见底，换题
4. 候选人表示不想继续 → 尊重边界，换题
```

**问题 B — 行为面试中识别不出 STAR 缺失**

修改 `modes.ts`：严格区分 Action 和 Result——

```
A（行动）："我拉了数据说服了他" → 这是 A，不是 R
R（结果）："A/B测试后DAU提升了5%" → 这才是 R
常见错误："我最后说服了对方" ≠ 业务结果
```

修复后验证：
- Case 4：Agent 从"你关注哪些指标？"改为正确换题
- Case 8：Agent 从"你怎么说服的？"改为"A/B测试的业务结果怎么样？"

### 轮次 2：解决 JSON 格式稳定性

**问题**：DeepSeek 偶尔输出不完整的 JSON（如 Case 2 两次测试中一次 JSON 截断）。

**修复**：
1. `lib/llm-client.ts` 增加 `jsonMode: true`，启用 `response_format: { type: "json_object" }`
2. `evaluation.ts` 简化输出格式指令，从约 50 行砍到约 25 行，去掉了 markdown 代码块包裹

修复后：全 14 用例零 JSON 解析失败。

### 轮次 3：扩展测试覆盖 + 补齐 MVP Prompt

新增：
- **3 个难度对比测试**（同一场景 × 三种难度）
- **3 个边界测试**（短回答、反问面试官、简历矛盾）
- **简历解析 Prompt**（`resume-parser.ts`）
- **复盘报告 Prompt**（精简版，JSON mode 兼容）

### 轮次 4：强化难度差异化

**问题**：友好/标准/压力三种模式的实际回复太过相似。

**根因**：`difficulty.ts` 描述了"语气"但缺少具体的、高频使用的签名句式。

**修复**：为每种难度定义必须使用的"签名句式"——

| 难度 | 签名句式 | 实际效果 |
|------|---------|---------|
| 😊 友好 | "没关系，不用着急" / "已经说得挺好的了" | "**没关系，不用着急。**能举个具体的例子吗？比如..." |
| 😐 标准 | "了解了。我想再深入问一下" / "能更具体一些吗？" | "具体来说，你主导的那3个增长实验是怎么设计的？" |
| 😤 压力 | "太虚了" / "你个人到底做了什么？" | "**太虚了。**你个人主导了3个增长实验，具体是哪3个？" |

三种模式现在肉眼可辨。

## 四、关键工程决策

### 为什么用 DeepSeek 而不用 Claude

| | Claude Sonnet | DeepSeek V3 |
|---|---|---|
| 输入 ¥/M tokens | ¥21 | **¥1** |
| 输出 ¥/M tokens | ¥105 | **¥2** |
| 一次全量测试(14用例) | ~¥2.5 | **~¥0.03** |
| 一次完整面试(12轮) | ~¥1.3 | **~¥0.06** |
| 100 人每天 3 次/月 | ~¥11,700 | **~¥540** |

**代价**：DeepSeek 的 JSON 格式一致性不如 Claude，通过 JSON mode 缓解。这是可接受的 trade-off。

### JSON mode 是关键

启用 `response_format: { type: "json_object" }` 后，14 个用例零格式错误。这是 DeepSeek 上跑结构化 Agent 的前提条件。

### Prompt 模块化的价值

- 改了 `difficulty.ts` 不会影响 `modes.ts`
- 每个模块的修改效果可以通过对比测试单独验证
- 未来换模型（比如换成 Claude），只需要改 `llm-client.ts` 的 provider 配置

## 五、后续建议

1. **生产环境可考虑混用**：高频面试轮次用 DeepSeek（便宜），复盘报告生成用 Claude（更细腻的反馈）
2. **增加真实面试录音测试**：目前是文本模拟，真实语音场景下 Agent 的表现需要另外验证
3. **Prompt 版本管理**：每次大改 prompt 时保留旧版本备份，方便回滚对比
4. **A/B 测试框架**：如果上线后有多个 prompt 版本，可以分流对比用户满意度
