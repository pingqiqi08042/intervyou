>
> - **产品成果**：产品已部署上线，支持语音输入、.docx 简历生成、用户自有 API Key。通过测试框架持续保证 Agent 行为一致性。完整项目文档（PRD + 技术方案 + Prompt 调优报告 + 终审报告）体现产品全流程思考。

## 面试叙事结构（STAR 完整版）

### S — Situation 背景
校招生准备面试时，最大的痛点不是缺题库——牛客、赛码都有题库。痛点是**没有人给他动态追问**。现有 AI 面试产品本质是固定题库+录音回放，不管候选人怎么回答，下一个问题都是预设的。真正的面试官会根据回答深挖细节。

另外，市面上的面试反馈也很泛——"表达能力 75 分"这种，候选人完全不知道怎么改。

### T — Task 任务
做一个能真正模拟面试官思维的产品：从简历出发，根据回答质量动态追问，给出可操作的改进建议。

### A — Action 行动

**1. 模型选型——为什么选 DeepSeek 而不是 Claude？**

我做了对比评测：

| 维度 | Claude Sonnet | DeepSeek V3 |
|------|-------------|-------------|
| 成本 | ¥21/M tokens | ¥1/M tokens |
| 中文能力 | 优秀 | 优秀 |
| JSON 格式 | 稳定 | 偶发不稳定（~5% 错误率） |
| 单次面试成本 | ¥1.3 | ¥0.06 |

**决策逻辑**：DeepSeek 便宜 98%——对于面试这种高频交互场景，成本差距巨大。JSON 稳定性的短板通过工程手段弥补：我设计了四层 Fallback 容错链路。同时保留 Claude 接入能力，用户可自行切换。

**2. Prompt 工程——4 轮迭代，每轮有数据支撑**

我建了一个 Prompt 测试框架——17 个用例覆盖开场、追问、换题、压力模式、边界场景。每次改 Prompt 之后跑一遍，看哪些 case 退化、哪些改善。

| 轮次 | 做什么 | 结果 |
|------|--------|------|
| V1 | 基础 Prompt 骨架 + 8 个测试用例 | 追问重复率 ~20%，Agent 常说"展开说说" |
| V2 | 加"强制换题信号"+ 防重复规则 | 重复率降到 5%，但 JSON 偶尔截断 |
| V3 | 开 JSON mode + 历史脱敏 + 压低 Temperature | JSON 错误率接近零，Token 消耗降 53% |
| V4 | 岗位自动识别 + 四层递进框架 + 三模式强化 | 17/17 通过，追问带具体方向不再泛泛 |

**3. 失败案例与收获**

**失败 1**：最初用 Temperature 0.7 + JSON 模式，发现 DeepSeek 经常输出不完整的 JSON（如 `{"message":"...` 就截断了）。排查后理解：Temperature 越高，模型输出越随机，JSON 结构的可靠性越差。解决方案：降到 0.5 + 开 JSON mode + 配合 retry，问题基本解决。

**失败 2**：上线后发现 Agent 会重复问"你具体负责了什么"——有时连续三轮问同一个问题。定位后发现问题不是 Prompt 写得不好，而是对话历史里包含了 Agent 之前的评分 JSON，LLM 看到自己的评分后行为锚定了。解决方案：每次发送前解析历史消息，只保留 `message` 字段，去掉所有评分数据。

**失败 3**：最初一版 Prompt 写了 3600 tokens，包含大量评分细则、字段说明、禁止规则。测试时发现 Agent 反而更容易出错——规则太多，它记不住重点。压缩到 1400 tokens（-53%）后，Agent 追问质量反而上升了。学到：Prompt 不是越长越好，关键是核心规则要放在最顶部。

### R — Result 成果
- 17 个测试用例全部通过，追问重复率降至零
- JSON 格式错误率从 5% → 接近零
- Token 消耗 3600 → 1400（-53%）
- 单次面试成本 ¥0.06（Claude 的 2%）
- 产品已部署上线，可在线访问

## 设计决策汇总（面试时展示思考深度）

| 决策 | 选项 | 选了 | 为什么 |
|------|------|------|--------|
| 问题来源 | 题库驱动 vs **简历驱动** | 简历驱动 | 简历驱动才能动态追问，题库驱动是死路 |
| 模型选型 | Claude vs **DeepSeek** | DeepSeek | 成本低 98%，JSON 不稳定性用工程手段兜底 |
| 追问框架 | 自由对话 vs **四层递进** | 四层递进 | 自由对话太不可控，四层递进保证追问质量和覆盖率 |
| 输出格式 | 纯文本 vs **JSON** | JSON | 需要内部评估数据（面试中不透露，复盘时释放） |
| 评估方式 | 人工审查 vs **测试框架** | 测试框架 | 人工太慢且不可重复，17 个 case 跑一轮只要 ¥0.03 |
| 数据库 | SQLite vs **PostgreSQL** | 本地 SQLite 开发，生产用 Neon PostgreSQL | 开发零配置，生产无缝迁移 |
| 部署 | 自建服务器 vs **Netlify** | Netlify | 免费托管 Next.js，自动 CI/CD |
