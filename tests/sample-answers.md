
**差回答**（太短、没信息量）：
```
我叫张三，浙大的，做过一些实习。
```

## 场景 2：字节项目深挖（简历深挖模式）

### 第 1 轮 — 入口问题

Agent 可能会问："你在字节做的推荐系统优化，具体做了什么？"

**好回答**（有细节、有技术）：
```
我们主要做三件事。第一是异步化，把原来同步等下游的调用改成用Kafka消息队列解耦，
这样推荐接口不用等下游全返回。第二是加了多级缓存，本地用Caffeine做L1，Redis做L2，
热点数据直接走缓存。第三是对MySQL做了索引优化，有几个慢查询走的全表扫描，加了
联合索引之后查询时间从2秒降到了几十毫秒。
```

**差回答**（笼统、无细节）：
```
就是优化了一下系统性能，改了一些代码，效果还不错。
```

### 第 2 轮 — 技术追问

Agent 追问（针对好回答）："异步化后数据一致性怎么保证的？"

**好回答**：
```
我们做了消息重试机制，下游失败了会重试三次。如果三次都失败就走兜底策略，
用默认推荐结果顶上。同时我们在Redis里维护了一个状态表，记录每个请求的
处理状态，防止重复消费。对于关键业务，我们没有异步化，保留了同步调用。
```

**差回答**（只有概念、无细节）：
```
就重试了几次，不行就算了。
```

### 第 3 轮 — 结果追问

Agent："你说延迟从200ms降到了80ms，这个数据是怎么测出来的？"

**好回答**：
```
我们在压测环境用JMeter模拟了5000并发，对比优化前后的P99延迟。
同时线上也做了灰度发布，先切10%流量观察了三天，P99从198ms降到76ms，
确认稳定后才全量。监控用的Prometheus+Grafana。
```

**答不上来的情况**（测试 Agent 换题）：
```
这个我不太清楚，好像是测试同学帮忙测的。
```

## 场景 3：网易项目（行为面试模式）

### 协作 / 冲突问题

Agent："你在网易实习时，有没有和团队成员意见不一致的情况？"

**好回答**（完整 STAR）：
```
有的。有一次讨论一个增长实验的方案，我觉得应该先做A/B测试验证效果，
但另一个同学想直接全量上，觉得数据已经够了。当时我是产品实习生，
他是正式员工，直接说"我不同意"肯定不太合适。

后来我拉了一些竞品数据，找了他们做类似实验的结果，基本都是先小流
量验证再放量。我把这些整理成一个文档，在周会上分享了一下。最后他
同意了我的方案，我们先做了两周的A/B测试，实验组的DAU确实涨了5%，
确认有效后才全量推的。

这件事让我学到：想说服别人，光说"我觉得"没用，要拿数据和案例说话。
```

**差回答**（只有 Action，缺 Situation 和 Result）：
```
有次我和同事意见不一样，后来我找了一些数据说服了他。
```

## 场景 4：技术面试模式

### 技术基础问题

Agent："你简历写了熟悉Redis，Redis的缓存淘汰策略你用过哪些？"

**好回答**：
```
我们主要用了allkeys-lru策略。因为推荐场景里热点数据变化比较快，
用了LRU能自动把不常访问的数据淘汰掉。另外我们对不同业务用了不同的
策略——用户画像数据用volatile-ttl设置过期时间，推荐结果缓存用
allkeys-lru。还监控了命中率，正常情况下在95%以上。
```

**差回答**（会引发深入追问）：
```
我们用的就是默认的淘汰策略，具体是什么不太记得了。
```

### 系统设计问题

Agent："你做的推荐系统如果用户量涨100倍，哪些地方需要重新设计？"

**好回答**：
```
首先缓存层要改。目前单机Redis扛不了那么大流量，需要做Redis Cluster分片，
或者加一层本地缓存减少网络开销。其次是数据库，目前MySQL单库，
读写分离和分库分表是必须的。然后消息队列要考虑堆积问题，Kafka的分区
策略和消费者扩容要提前规划。最后是推荐算法本身，目前是规则+简单模型，
流量大了可能需要上实时特征和深度学习模型，那就要加特征平台和模型服务。
```

**差回答**：
```
加机器扩容就行了。
```

## 场景 5：压力模式专属回答

> 选「压力」难度时，Agent 语气会更犀利。以下是应对示例。

### 数据质疑

Agent："15%这个数字怎么来的？我怎么知道不是你编的？"

**好回答**：
```
我们从Amplitude拉了实验前后的DAU数据。实验组5万人，对照组5万人，
实验周期14天。实验组DAU从日均102万涨到117万，涨幅14.7%四舍五入
就是15%。对照组同期从101万涨到103万，排除了自然增长的影响。p值
小于0.05，统计显著。
```

**顶不住的情况**：
```
这个...具体的数据我现在记不太清了，当时是数据分析同事帮忙算的。
```

### 归因挑战

Agent："你一直说'我们团队'——你个人到底做了什么？"

**好回答**：
```
实验设计和数据分析是我独立完成的。我写了实验假设文档，跟工程团队
沟通了埋点需求，自己用SQL跑了实验数据，做显著性检验也是我做的。
产品方案是leader给的方向，但具体的实验执行和数据分析是我做的。
```

## 测试技巧

| 想测试什么 | 怎么回答 |
|-----------|---------|
| Agent 追问能力 | 用"好回答"——Agent 应该深入追问具体技术点 |
| Agent 换题能力 | 连续 2 轮说"不太清楚"——Agent 应该主动换题 |
| 压力模式 | 选压力 + 给模糊回答——Agent 应该犀利质疑 |
| 友好模式 | 选友好 + 说"不会"——Agent 应该给提示引导 |
| STAR 检查 | 给缺 Result 的回答——Agent 应该追问结果 |
| 语音输入 | Chrome 下点击麦克风说话 |
